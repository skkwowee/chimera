# 2026-06-14: NLA RECONSTRUCTION OBJECTIVE added to the bridge design
# ============================================================================
# WHAT: closed the latent->text bridge into a Natural-Language Autoencoder by
#   adding a TEXT-ONLY DECODER (reconstructor) that rebuilds the frozen world-
#   model latent from Qwen's GENERATED reasoning text. The existing bridge
#   (latent -> resampler -> soft tokens -> Qwen) is the ENCODER half; the new
#   decoder is the DECODER half. Target = the pooled-512 vector z=h.mean(dim=2)
#   that value_head actually reads (grid is [B,L,11,512], slots identity-fixed
#   so no Hungarian matching). Loss = (1-cos) + 0.1*MSE on standardized z.
#   HARD FIREWALL: the decoder consumes the decoded answer STRING re-tokenized
#   fresh ONLY -- zero tensor path from latent/soft-tokens/residual-stream, or
#   reconstruction is trivial cheating (enforced by assertion + unit tests;
#   shuffled/empty text must score at the latent-mean floor).
# WHY: two things the bridge was missing.
#   (A) FAITHFULNESS METRIC, stronger than ablate-the-latent: reconstruct z
#       from Qwen's generated text and measure fidelity (cosine / variance-
#       explained vs shuffled-text, latent-mean, and ablated-bridge controls,
#       plus value-head-agreement + readability guard). Gives a NUMBER for "is
#       Qwen faithfully rendering the latent or hallucinating past it" -- the
#       embellishment risk ablate alone is blind to. Ablate and recon are
#       COMPLEMENTARY (ablate = latent is USED; recon = text PRESERVES it);
#       recon replaces nothing.
#   (B) LABEL-FREE training/bootstrap signal: trained on reconstruction
#       fidelity, NO answer key -- sidesteps "where do correct descriptions
#       come from / is the teacher LLM stable on raw ticks" (it isn't; that's
#       why the world model exists).
# HONEST CAVEATS (do not lose): recon-faithful != good tactical reasoning (NLA
#   optimizes info-preservation, can yield faithful-but-stilted text) -- so it
#   COMPLEMENTS, never replaces, templated grounding (format/vocab) and GRPO
#   (reasoning quality, value-through-rollout stays sole quality signal).
#   Capacity is open: bounded text vs 512-d (x11) latent -- target the
#   decision-relevant subspace, not the raw vector.

#   FROZEN held-out verbalizer outputs (joint training -> private cipher, the

## What chimera is now
A next-state-prediction WORLD MODEL for CS2: a causal spatiotemporal
transformer over game-state frames, trained state -> state (NO text), like LM
pretraining but over game states. Value, events, and (phase 2) language are
heads/bridges on the learned latent.

## Why we pivoted (REAL findings from the VLM line — do not lose these)
- Round encoder SATURATES at ~16 demos (negative slope beyond). Outcome
  supervision is ~1 bit/round — information-starved.
- Change-point losses find STATISTICAL, not SEMANTIC, boundaries — wrong
  objective for "events".
- Claude captions are CIRCULAR: only paraphrase the structured features fed in
  (discriminative-check gap only +0.008 over the structured ceiling).
- Commentary grounding: global VOD<->demo alignment locks at 4.6 sigma
  (kills<->caster-name xcorr) but per-event anchoring only ~25% (auto-caption
  ASR name-recall ceiling). PARKED to phase 2.
The pattern: every signal was sparse/downstream/circular. Next-frame
prediction is dense, self-supervised, grounded in real dynamics. That's the bet.

## What changed this session (2026-04-17 → 04-18) — GRPO speed fix

Took the failed F06 run from 14h/40 steps (~21 min/step) down to ~5-7 min/step
on the same H200 pod, before attempting prefix sharing for the next 20-30%.

### Root cause of the 14h slowness (diagnosed)
- Pod's torch was built for CUDA 13.0 but pod's nvcc was 12.4 → causal-conv1d
  and flash-linear-attention wheels couldn't link → transformers fell back to
  Python ref impl. Torch inductor then tried to JIT-rescue the fallback path
  (~30 min compile storm with 35 workers, never recovered perf because of
  graph breaks on dynamic MoE expert routing). Net: 14h for 40 steps.
- A second-order issue compounded it: with peft, gradient_checkpointing kept
  the "use_cache=True is incompatible with gradient checkpointing" warning
  firing every generate() call → KV cache off → O(n²) decode per token.

### Fixes applied (commits 129456c → 3eae3e4)
- src/training/grpo_trainer.py:
  - _preflight_kernels(): hard-fail at load_model() if causal_conv1d or
    flash_linear_attention can't import. Aborts in ~10 sec instead of 14h.
  - attn_implementation="flash_attention_2" default + --attn-impl CLI flag.
  - gradient_checkpointing=False default (was unconditionally on). Since
    H200 has 144GB and the model is 65GB, we have ~80GB activation headroom
    — memory isn't the bottleneck. Removing GC fixes the use_cache warning.
  - _set_generation_mode(): walks PeftModel.get_base_model() to set use_cache
    on the base model config (which the wrapper does NOT propagate to).
- scripts/pod_setup_grpo.sh (new): venv-based pod bootstrap. Iteratively fixed:
  - PEP 668 blocks `uv pip install --system` on Ubuntu 24.04 → use uv venv
    with --system-site-packages.
  - `uv venv` doesn't ship pip → use `uv pip install --python $VENV_PY`.
  - Transitive deps (transformers, trl, peft) auto-upgrade torch to cu130
    even with --system-site-packages → pre-pin venv torch to system version
    via pytorch.org wheel index.
  - PyPI dist name is `flash-linear-attention`, not `fla` (only the import
    name is `fla`).
  - Wire CUDA_HOME=/usr/local/cuda + PATH so causal-conv1d build finds nvcc.
  - Strict CUDA major check was a false positive (nvidia-smi "CUDA Version"
    is the max-supported by the driver, not the installed runtime). Loosened
    to warn-only.
- scripts/run_grpo_pod.sh (new): launcher with TORCHDYNAMO_DISABLE=1
  (skips the ~30 min inductor compile storm), kernel re-verify, nohup.
- requirements.txt: added faiss-cpu (RECALL needed it; was crashing the
  speed test before model load).

### Why `mean_strategy` = -0.031 (the actual experimental finding)
Audited RECALL end-to-end (src/training/recall.py) against f08v4_resumed
useful_jumps.jsonl. The negative strategy score is not a sign the model's
advice is bad -- it's a sign the RECALL signal-to-noise is too low to
judge. Three structural issues:

1. Action embedding is 5-dim. movement_direction (-1/0/1),
   objective_direction (-1/0/1), utility_count/4, initiated_engagement
   (0/1), engagement_delay (0.2 or 0.8 binary). Cardinality ~54 buckets
   at best. Cosine similarity with threshold 0.5 on 5 dims is very
   coarse -- tiny keyword-count differences flip which neighbors get
   "matched".

2. Action extraction from free-form advice uses keyword counts against
   ~20 English keywords (_extract_action_from_text). Model natural
   phrasing variation dominates the feature value; two tactically-
   identical advice blocks produce different action vectors.

3. Asymmetry: state embedding is 19-dim from demo ground truth (dense,
   deterministic, fine) but action embedding is 5-dim from text regex
   (sparse, stochastic, weak). kNN matches on rich state, then filters
   by a noisy action vector. The filter is where the signal dies.

Result: Q_hat ~ V_hat + noise, advantage ~ noise, reward ~ noise. 100
GRPO steps over noise gave a small negative bias (-0.031) but no
meaningful content direction.

### Diagnostic added (scripts/recall_diagnostic.py)
Standalone, runs offline. Loads a saved LoRA + RECALL index, generates
advice for N val samples, for each generation dumps the full kNN
internals: extracted_action dict, action_vec + norm, state_vec_norm,
k_used, v_hat, q_hat, advantage, confident flag, matched_count,
cos_sim min/max/mean, cos_sim_above_threshold_frac, fallback_mode,
top5_cos_sim. One row per generation to
/workspace/outputs/diag/recall_diag.jsonl.

Goal: before investing in a richer state/action/scoring triple, confirm
with real numbers that the current one is in the noise-dominated regime.
Expected result (hypothesis): most samples will have cos_sim_mean near
the threshold, matched_count varying wildly, |q_hat - v_hat| comparable
to the std over neighbor outcomes (i.e. the signal is within the
baseline's own noise).

### Research framing (from this session's discussion)
This is genuinely novel territory. Closest analogues:
  - Chess/Go: discrete action space, exact outcome. Not our situation.
  - RLHF: human preference pairs. We don't have pairs of advice blocks.
  - Code RL: executable unit tests. We have demos, not tests.
Natural-language strategic advice scored against demo outcomes via kNN
has no clean precedent we've found. The experimental unit for iteration
is a (state-representation, action-representation, scoring-function)
triple, and the metric we actually care about mid-iteration is the
signal-to-noise of the advantage estimate, not just "did the model
learn" -- because if the signal is noise, the model will converge to
whatever local minimum minimises KL drift.

### Concrete next directions (in increasing scope)
(a) Quick: run recall_diagnostic.py against f08v4_resumed to confirm
    the noise hypothesis empirically. One short pod session.
(b) Medium: richer action representation. Instead of 5 coarse floats
    extracted from keyword counts, parse structured keys directly out
    of advice (e.g. advice.target_site, advice.weapon_priority,
    advice.timing, advice.coordination). Each becomes a dim that keeps
    its meaning. State stays at 19 dims or grows with it.
(c) Big: replace kNN-over-keyword-action scoring with a Claude-judge
    that scores advice plausibility against the game_state on an
    explicit rubric. Expensive per-sample but trustworthy. Plausibly
    the "correct" answer long-term.

Lean MEDIUM first: cheapest thing that might move the signal floor
above the noise floor. If diagnostic + medium iteration confirm we can
get real signal, the big-scope judge is a follow-on, not a prereq.

### Diagnostic: within-group variance on f08v4 + f08v4_resumed
Ran a within-group variance analysis on the existing useful_jumps.jsonl

The question: when 2+ completions pass the format gate at the same step,

  reward_std < 0.01:   68/202   (33.7% effectively no signal)
  reward_std < 0.05:  178/202   (88.1% weak signal at best)
  format_passes mean   1.86/4
  Among-passers spread (RECALL's real job):
    2/4 passing (77 steps): MEDIAN spread = 0.000  (identical scores!)
    3/4 passing (38 steps): median 0.0154
    4/4 passing (7 steps):  median 0.0154

f08v4_resumed (200 steps):
  reward_std < 0.01:   85/200  (42.5%)
  format_passes mean   2.75/4   (model learned format better post-resume)
  Among-passers spread:
    2/4 passing (52 steps): median = 0.000
    3/4 passing (85 steps): median 0.0101  (50% < 0.01)
    4/4 passing (43 steps): median 0.0154

### What this proves
The reward variance we DID observe at the top level was dominated by the
format gate: "the one completion that passed got ~0.02, the three that
failed got 0." When we filter to ONLY the passing completions -- exactly
where RECALL is supposed to do strategic discrimination -- the median
spread is 0.000 for the most common case (2/4 passing) and ~0.01 for the
rest. RECALL was giving identical or near-identical scores to different
strategic advices in the same state.

So the F08v4 model learned "emit valid JSON" (format_passes 1.86 -> 2.75)
but did NOT learn strategic content from RECALL, because RECALL was not
a strategic-content signal in the first place. The 19-dim state vector
is dominated by surface features (map one-hot, side, HP) so kNN retrieves
mostly same-round neighbors, V_hat == Q_hat, advantage == 0.

This matches the structural argument from the prior diagnostic but now
it's empirical on real training, not hypothetical.

### Decision: switch reward signal to Claude judge
We chose option (c) from the prior session's options instead of (b).
Reasoning: the state-equivalence problem in CS2 is fundamentally hard
because state similarity is a decision-function property, not a
geometric one. No fixed embedding -- hand-engineered or LM-based --
captures tactical equivalence well enough with our limited data
(4 demos, 80 rounds).

The judge route bypasses state-equivalence entirely. Per-step:
  1. Generate G=4 completions
  2. Single judge call with state + 4 completions + rubric
  3. Judge returns scores 0-1 per completion (or pairwise rankings)
  4. Use judge scores in place of recall_reward
  5. Format gate stays multiplicative as before

Cost: ~$0.50-2 per 100-step run with Opus 4.7. Trivial vs pod cost.

Caveat: the judge's notion of "good CS2 advice" is another model's prior,
not actual coach labels. Mitigated by including pro_action and round_won
in the judge context so it can anchor on "did the advice match what the
pro did, who then won/lost."

### What the new judge_reward needs
- Inputs: state (game_state JSON, pro_action, round_won) + list of G
  completion strings.
- Outputs: list of G floats in [0, 1]. Returns one value per completion.
- Single Anthropic API call per step (not per completion).
- Robust JSON parsing -- if the judge's own response is malformed, fall
  back to all 0.5 (neutral, no gradient that step).

### Concrete implementation plan
- Add anthropic to requirements.txt (probably already there for baseline_eval).
- New module src/training/judge_reward.py with single function judge_reward(
    state_dict, completions: list[str]) -> list[float].
- Add "judge" mode to REWARD_MODES in train_grpo.py:
    [perceptual_accuracy_reward, judge_reward], weights [0.20, 0.80]
  Same shape as "recall" mode -- swap recall_reward for judge_reward.
- The judge call has to happen at the GROUP level (one call sees all G
  completions), not per-completion. This is a structural change in the
  manual GRPO loop -- right now reward_fns are called per-completion in
  a list comprehension. Either refactor the loop or have judge_reward
  cache its scores keyed on (state_id, completion_hash) so per-completion
  calls hit the cache after the first.
- Caching is the cleanest minimal-diff approach: judge_reward(state, comp)
  computes scores for ALL G completions on first call, caches keyed on
  (state_id, set of completion hashes), returns the score for the one
  completion that was passed.

### Test-against approach (deferred, not blocking)
Build a small "pseudo-gold" dataset (30 states x 4 advices A/B/C/D where
A is constructed-correct, B constructed-wrong, C generic, D plausible
but wrong) to compare any future scoring function head-to-head. Synthetic
labels by intuition, not real ground truth, so it can only validate
"is this scorer better than chance" not "is this scorer right." Useful
for differential comparison between candidate scorers; not blocking the
judge implementation.


### Glossary clarification: what `reward_std` measures

For each GRPO training step, the model generates G=4 different completions
in response to the SAME game state (sampling diversity from temperature>0
is what produces the variety -- they're four different pieces of strategic
advice for one situation, e.g. for a 2v3 post-plant: "hold crossfire" vs
"push connector" vs "play smart" vs "rotate ramp").

Each of the 4 gets a reward score. `reward_std` is the standard deviation
across those 4 scores. GRPO's whole game is `advantage_i = (r_i - mean_r)
/ (std_r + eps)` -- the relative ranking within the group is the gradient
signal. So:

  - reward_std HIGH (e.g. 0.18 across [0.7, 0.2, 0.1, 0.6]):
    scoring function differentiated the strategic content. Gradient
    pushes model toward the high-scoring advice style.
  - reward_std NEAR ZERO (e.g. 0.0 across [0.04, 0.04, 0.04, 0.04]):
    scorer gave essentially the same score to all 4 different advices.
    Advantages collapse to noise; no useful gradient.
  - reward_std MODERATE BUT FORMAT-DRIVEN (the f08v4 case, e.g. [0.04,
    0, 0, 0]): variance comes entirely from "one passed the format
    gate, three didn't." The model learns format discipline, not
    strategy. The strategic scoring component contributes nothing.

The diagnostic that nailed RECALL was filtering to ONLY the completions
that passed the format gate and asking "what's the spread *among them*?"
Answer for the most common case: 0.000. Identical scores for genuinely
different strategic advices.

2026-05-16 (cont. 4) — v9 change-point: mechanism works, objective misaligned
================================================================================

User asked for "tick aggregation TRAINED" — model learns its own event
windows, not fixed 2s. Approach C (change-point segmentation) chosen
over A (mixture-of-horizons) for being closer to the "learn what an
event IS" goal.

DESIGN
- ChangePointHead (~0.4M params on top of v6 encoder):
  - per-tick boundary head → b_logits → sigmoid → b_prob
  - training: Bernoulli sample b_hard ~ p; STE for gradient through hard
  - eval: top-K positions where K = target_density × T (reproducible)
  - cumsum(b_hard) → segment ID per tick
  - per-segment scatter_mean of h → forecast head → 6 aggregates
    (kill_t count, kill_ct count, plant/defuse/explode/round_end flags)
- Loss:
  - L_segment = smooth-L1 on log1p(counts) + BCE on flags
  - L_density = (mean(b_prob) - target_density)^2  (around-target, not minimization)
  - target_density = 0.02 (= 1 boundary per 50 ticks ≈ 6.25s, ~20 segments/round)
- Init: boundary head bias = logit(target_density) ≈ -3.9, so model
  STARTS at target density, doesn't collapse to 0 boundaries while
  hunting for the right rate
- (Earlier attempt with min(b_prob) penalty + threshold>0.5 collapsed
  immediately to b_prob≈0 / 1 segment per round. Fixed with the around-
  target MSE + Bernoulli sampling + bias init.)

RESULTS (v9 50ep on feature_schema_v2 81-demo split)
- val_acc_event_only best 0.5370 (similar to v6 0.5269, v7 0.5391)
- L2-G2 probe 0.7694 (slight best across versions)
- L_cp_count 0.04, L_cp_bin 0.025 — forecast head training fine
- cp_density stays locked at 0.020 throughout
- cp_n_segments_avg = 20-23 per round on val

THE NEGATIVE FINDING (visualize_changepoints.py)
Plotted per-round b_prob with picked boundaries (green) and actual events
(colored dotted lines from kills/plants/defuses). Boundaries cluster in
1-2 tight groups per round rather than spreading across events.

Quantitative: fraction of actual events within ±W ticks of a learned
boundary (val, 6 rounds, 46 total events):
  ±2 ticks: 4/46 = 8.7%   (random baseline ~12.5%)
  ±4 ticks: 5/46 = 10.9%  (random baseline ~22.5%)
  ±8 ticks: 6/46 = 13.0%  (random baseline ~42.5%)

Boundaries are ANTI-CORRELATED with events. The model found a
degenerate equilibrium: cluster boundaries in quiet periods (where
predicting "0 events per segment" is trivially correct), then absorb
all action into one big segment. The per-segment forecast loss does
NOT push toward event-aligned boundaries.

IMPLICATIONS
1. "Learn event segmentation via per-segment count forecasting" is a
   misaligned objective. Even though density + forecast loss converge,
   the boundary positions are useless for the original goal (learning
   what's tactically interesting).
2. To actually learn event boundaries, the loss must reward
   homogeneity WITHIN segments, not just predictability across
   segments. Candidates:
   - Contrastive: ticks in same segment should be similar in embedding,
     across-segment ticks should differ
   - Per-segment outcome prediction (round_won) — outcomes change at
     events
   - Intra-segment variance penalty on raw state features (segments
     should be "stable" stretches)
3. For paper: useful methodology-level negative — shows that "learning
   events from forecasting" needs a sharper objective than count-
   matching.

Background: v9 (cont. 4) entry said v9 boundaries were anti-correlated
with events based on 6 val rounds (46 events). That was a SAMPLING
ARTIFACT. On the full val set (262 rounds, 1918 events) v9 actually
shows +10.8% / +7.1% / -1.7% alignment lift at ±2 / ±4 / ±8 ticks
vs random. Earlier doc is corrected by THIS entry.

Hypothesis going in: per-segment count forecasting (v9) is misaligned;
"intra-segment homogeneity" objectives should produce cleaner event-
aligned boundaries.

v10 DESIGN (one new TrainConfig.change_point_loss flag + optional aux):
  forecast   — v9 baseline (per-segment event counts: kills, plant, etc.)
  contrastive — InfoNCE; same-segment ticks pull together in h-space
  variance   — intra-segment k-means-style variance penalty on h
  outcome    — same head as forecast but predict NEXT segment's counts

Plus a new optional aux term `change_point_aux_align_weight`:
  L_aux = -E[ b_prob[t] * detach(||h[t]-h[t-1]||) ]
  Idea: give DIRECT gradient to boundary head telling it to fire where
  the encoder state changes a lot. (v9 had no such direct gradient —
  seg_id is detached after STE, so boundary head only learned from the
  density penalty.)

Six new training runs (50 epochs each, d_model=512, n_layers=4, same
v6 config as v9):
  v10a_contrastive       (aux=0.1)
  v10b_variance          (aux=0.1)
  v10c_outcome           (aux=0.1)
  v10a_contrastive_noaux (aux=0.0)
  v10b_variance_noaux    (aux=0.0)
  v10c_outcome_noaux     (aux=0.0)

NEW DIAGNOSTIC (scripts/compare_changepoint_variants.py)
Evaluates each ckpt over the full 262-round val set:
  - actual events tagged: kill_t, kill_ct, plant, defuse, explode, round_end
  - picked boundaries = top-K positions by b_prob, K = target_density * T
  - alignment[W] = % of events within ±W ticks of any picked boundary
  - random baseline[W] = 1 - (1 - (2W+1)/T)^K   (uniformly-placed boundaries)
  - lift = alignment - random; lift > 0 means boundaries beat chance

  variant                  avg_segs   W=2 lift   W=4 lift   W=8 lift
  --------                 --------   --------   --------   --------
  v9 forecast (baseline)      17.9     +10.8%      +7.1%      -1.7%
  v10a contrastive (aux)      29.2      -9.6%     -16.6%     -29.1%
  v10b variance    (aux)      28.7      -6.5%     -13.0%     -25.2%
  v10c outcome     (aux)      26.5      -9.5%     -16.5%     -29.0%

  v10c outcome     (no aux)   18.4      -6.6%     -13.2%     -25.0%

All v10 variants have val_acc_event_only within 0.005 of v9 (0.5370):
  v10a 0.5254 / 0.5263, v10b 0.5386 / 0.5369, v10c 0.5391 / 0.5367.
Encoder quality is preserved; only the boundary placement differs.

NEGATIVE FINDING (strong)
The intra-segment homogeneity hypothesis is WRONG for this domain. All
three "homogeneity" losses produce boundaries that are anti-correlated
with events:
  - At ±2 ticks: v9 hits 20.3% of events vs 0-3% for v10 variants.
  - At ±8 ticks (loose): v9 nearly random (27% vs 29% baseline), but
    v10 variants are systematically AVOIDING events (1-4% vs 29%).

The aux state-change term made things WORSE: avg_segs ballooned to 26-29
(density target was 0.02 → expected ~17). Aux drives boundaries to
high-||delta h|| points; events ARE high delta h, but so are many non-
event positions (weapon switch, money awarded, freeze→playing
transition). Aux spreads boundaries across all those noisy state-change
points, pushing them AWAY from events on average.

WHY V9 WINS (interpretation)
Per-segment EVENT FORECASTING has a structural property the other losses
lack: it forces each segment's pooled representation to predict the
event mix WITHIN that segment. Segments that straddle real event
boundaries (e.g., span a kill phase and a plant phase) get a confused
count target and high forecast loss. Segments aligned with natural
phases get clean targets and low forecast loss. So the loss implicitly
rewards event-aligned boundaries via the encoder's gradient on h —
even though seg_id itself is detached.

The other losses don't have this property:
  - Contrastive/variance reward h-space smoothness. Encoder h is
    dominated by player position, equipment, time — events are RARE
    perturbations. Smoothness loss prefers segments inside long
    "quiet" stretches and places boundaries to ISOLATE the noisy
    event ticks (one boundary just before, one just after) rather
    than ON events. Net: -25% alignment at W=8.
  - Outcome (next-segment forecasting) was supposed to inherit
    forecast's alignment property but applied to the NEXT segment.
    But the model can predict the next segment's events from the
    current segment's content regardless of where the current
    segment ends — there's no penalty for misaligned current
    boundary as long as the encoder learns enough about "what
    typically follows this kind of state". Lift = forecast's lift.

PUBLICATION-LEVEL TAKEAWAY
This is a useful methodology contribution: the intuition that
"homogeneity-based losses produce cleaner segments than predictability-
based losses" — which has support in the unsupervised segmentation
literature for static signals (images, audio) — does NOT transfer
to game state where the "homogeneity" axis is dominated by ambient
state change (player movement, equipment) rather than event-relevant
state change. The forecast loss's implicit alignment is the better
inductive bias for this domain.

The earlier (cont. 4) note proposing contrastive / variance / outcome
as fixes was wrong; v9 forecast IS the right loss, the original 6-
round sample just under-estimated its alignment.
